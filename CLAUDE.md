# Space Invaders Mobile — Claude Code Instructions

## What is this project

A mobile remake of a Space Invaders clone originally built in Unity/C#. The core problem of the original was manual difficulty calibration — 8 parameters per level, adjusted by hand. This project solves that with an automatic level generation system, a visual map editor, and Firebase analytics to detect problematic levels.

The architecture is designed to be reusable: the Level Engine, Monetization Plugin, and Analytics Plugin are framework-agnostic TypeScript packages that can power future 2D games without modification.

## Stack

```
  Mobile Game:    Expo ~52 + react-native-skia (Android first, iOS after)
  Level Engine:   TypeScript pure — zero native deps, testable with Jest
  Calibrator:     Next.js 14 + HTML5 Canvas (dev tool, localhost:3001)
  Monetization:   AdMob (rewarded ads) + RevenueCat (IAP)
  Analytics:      Firebase Analytics
  OTA Updates:    EAS Update (publish level changes without store review)
  Build:          EAS Build (cloud iOS/Android builds)
```

## TDD — Mandatory

No feature ships without passing tests.

```
1. RED    → write the failing test first
2. GREEN  → implement the minimum to pass
3. REFACTOR → improve without breaking
```

- Run the test to confirm it **fails** before implementing
- Run again to confirm it **passes** after implementing
- Commit

Tests live in `src/__tests__/*.test.ts` within each package. Never write implementation before a failing test.

## Architecture

```
  Game Loop (pure TS) → IRenderer → SkiaRenderer   (mobile/GPU)
                                  → CanvasRenderer  (calibrator/browser)

  LevelEngine.generate(request)
    → CalibratorStrategy  →  CurveCalibratorStrategy   (MVP)
                          →  SimulationCalibratorStrategy (v2)
                          →  TelemetryCalibratorStrategy  (v3)
    → EntityRegistry      →  registered entity types
    → LevelDefinition
```

### Principles

- **Renderer-agnostic game logic** — physics, collision, AI, and the game loop live in pure TypeScript. They have zero knowledge of Skia or Canvas. Only `IRenderer` interface is injected.
- **Dual renderer** — `SkiaRenderer` (mobile) and `CanvasRenderer` (browser/calibrator) implement the same `IRenderer`. What runs in the browser during calibration is exactly what ships to mobile.
- **Registry pattern** — the Level Engine doesn't know "Space Invaders". It knows entity types registered at startup. This makes the entire platform reusable for future 2D games.
- **Pluggable calibration strategies** — swap calibration algorithm by injecting a different `CalibratorStrategy`. The game code doesn't change.
- **OTA-first levels** — `levels.json` is generated by the calibrator and published via EAS Update. No store review needed for level changes.
- **Packages are pure TypeScript** — `packages/*` must have zero native dependencies. They are testable with Jest on any machine without a simulator.

## Git Workflow — Small PRs, Granular Commits

**Max ~15 files per PR. One concern per commit. Never mix sprints or packages.**

### PR Split per Sprint

**Packages (`packages/*`):**
1. `types` — interfaces and type definitions
2. `domain` — core implementation
3. `test` — all tests

**Apps (`apps/game`, `apps/calibrator`):**
1. `chore/infra` — deps, config, renderer setup (one-time)
2. `feat` — feature screens/components
3. `test` — all tests

**Rules:**
- Branch from `main` for each PR; merge sequentially
- Each PR must be independently reviewable
- **NEVER push directly to `main`** — all changes go through a PR

## What NEVER to do

- Do not add native dependencies to `packages/*` — they must stay pure TypeScript
- Do not modify `IRenderer` without updating both `SkiaRenderer` and `CanvasRenderer`
- Do not edit `levels.json` manually — it is always generated by the calibrator
- Do not call `LevelEngine.generate()` before `registerEntityType()` calls complete
- Do not add features beyond what was asked
- Do not refactor code you are not modifying
- Do not commit tests with `.only` or `.skip`
- Do not push directly to `main`
- Do not hardcode level parameters — they flow from the calibrator → levels.json → EAS Update

## Gotchas

- **Native deps in packages**: adding any native module to `packages/*` breaks the calibrator (browser) and Jest tests — always check `package.json` before installing
- **IRenderer interface drift**: if `SkiaRenderer` and `CanvasRenderer` diverge, the browser calibration no longer matches mobile behavior — tests must cover both renderers
- **registerEntities order**: entity types must be registered before the engine generates any level — call `registerEntities()` at app startup, before navigation
- **EAS Update scope**: OTA updates only work for JS/TS/assets changes. Adding a new native module (e.g. a new Expo SDK feature) requires a full EAS Build + store submission

## Best Practices

1. Read the spec at `docs/superpowers/specs/2026-04-18-space-invaders-design.md` before implementing any feature
2. TDD on every feature — no exceptions, even for "simple" changes in the Level Engine
3. Keep `packages/*` free of native deps — if it can't run in Node.js + Jest, it doesn't belong there
4. Test both renderers when changing game logic — behavior must be identical in Skia and Canvas
5. The calibrator is a dev tool, not a product — optimize for developer speed, not polish
6. Use `FreeMonetization` in dev and tests — never mock AdMob or RevenueCat flows directly
7. Use `NullAnalytics` in dev and tests — Firebase events should only fire in production
8. Small PRs — max ~15 files, one concept per commit, never mix packages or sprints
9. Never push directly to `main`
10. `levels.json` is the contract between the calibrator and the game — treat it as an API
11. Survival mode generates levels at runtime — it must never block the game loop (keep generation fast)
12. When adding a new entity type, register it in both `registerEntities.ts` (game) and the calibrator toolbox
